# 作業規約（全プロジェクト共通）

過去の会話で繰り返し指摘された内容のルール化。プロジェクト固有のCLAUDE.md/AGENTS.mdやスキルと矛盾する場合はそちらを優先する。

## 言語
- 会話・コードコメント・commitメッセージ・PR本文・テーブルやカラムのdescription・ドキュメントはすべて日本語で書く。
- 例外は、コード識別子・技術用語・Conventional Commitsのprefixと、ユーザーが明示的に英語を指定した場合のみ。セッションが長くなっても、別エージェントへの引き継ぎ後も維持する。

## スコープ規律
- 指定されたファイル・箇所だけを変更する。最小diffを徹底し、指定外のリファクタや「ついで」の変更をしない。
- 指示にないロジック・条件・データを発明しない。補完が必要なら「◯◯は指示になかったので△△と仮定した」と明示する。
- 既存のコメント・設定・ユーザーが書いた文章を勝手に消したり書き換えたりしない。「typo以外一文字たりとも改変しない」系の指示は文字通りに守る。

## 訂正への対応
- 修正指示を受けたら、推測で直し始める前に元の指示と訂正文を読み直し、何を誤解したかを特定してから手を動かす。同じ誤解のまま2回目の修正をしない。
- 「◯◯を参考に」と参照実装を指定されたら、必ず先にそれを読む。
- 一度伝えられた規約・指示（「毎回◯◯すること」等）はセッション終了まで維持し、引き継ぎ文書にも書き残す。

## 要件の完遂
- 複数の要件はチェックリスト化してから着手し、完了報告は項目ごとに書く。副次要件（「本件とセットで〜」）を落とさない。
- 質問の対象が複数あれば、すべてに答える。

## 裏取り
- データは実際にクエリを投げて、コードの挙動はコード本体を読んで、外部仕様は公式情報を調べてから断言する。
- ソースにない事実を作らない。確認できないまま答えるときは推測であることを明示する。

## コードコメント
- 「何をしているか」ではなく「なぜこうしているか」を、処理ブロックごとに日本語で書く。特にSQL・CI設定・シェルスクリプト。求められる量はかなり多め。迷ったら書く。

## ドキュメント・コメントは読み手（人間）本位
- 作業の中間過程を成果物に残さない。「変更1は廃止して変更2にした」のような、会話の経緯を前提とした記述は読み手には無意味。常に最終状態だけを書く（経緯そのものが主題のADR等は例外）。
- AIエージェント向けの注意書き調の文（「〜は書かないこと」等のプロンプト風記述）を人間向けドキュメントに混ぜない。
- 脳死で箇条書きしない。単なる羅列は読み手の負荷が重い。構造化できる場合はテーブル形式や、構造化した上での箇条書きにして、読み手が理解しやすくなるための努力をする。
  - 複数の比較観点がある場合は pros/cons でまとめず、観点ごとの相対的な優劣を ◎→○→△→× や ★★★→★★☆→★☆☆ 等で段階的に示し、補足を添える。

    ```markdown
    | パターン概要 | 採用 | 観点1 | 観点2 | 備考 |
    | :--- | :--- | :--- | :--- | :--- |
    | A | ✅ | ◎<br>補足 | △<br>補足 | |
    | B | | ○<br>補足 | △<br>補足 | xxx |
    ```

  - テーブルに収まらないほど情報が多い場合は、「比較対象の概要と結論 → 観点ごとのセクション」のようにセクション自体を分けて構造化する。

## 文章の推敲
- 文章への部分修正を数回重ねたら、依頼されなくても最後に全体を通しで読み、日本語の使い方・論理展開・重複・トーンの整合を確認して直す。指摘の列挙で終わらせず修正まで行う。
- 1文に3要素も詰め込まない。濃淡をつけて文を分ける。

## Git・PR・成果物
- commitはConventional Commits形式のprefix + 日本語本文。commit・push・PR作成は迷わずまっすぐ実行する（破壊的操作・force push類は例外で、必ず確認する）。
- リネームやリファクタの後は、旧名への参照が残っていないか全参照を確認してから完了報告する。マージ・削除系の作業後も、実際に反映されたかを確認する。
- 生成物・一時ファイルはリポジトリ直下に散らかさず、専用ディレクトリにまとめる。
- ワークツリーやブランチの掃除を頼まれたら、対象を漏らさず全部片付ける。

## セキュリティ
- `.env`・credential類は明示的な許可なく読まない。機微な情報源へのアクセスは事前にユーザーへ確認する。
- 秘匿情報（private key本文、passphrase、トークン等）をドキュメントやPRコメントに書かない。

## SQL / Dataform
- ref/sourceを読むだけのdeclaration用CTEは `select * from ${ref(...)}` のみとし、絞り込み・カラム加工は後段のCTEで行う。declaration用CTEはコード上部にまとめる。
- boolean条件は `= true` / `= false` ではなく `xxx` / `not xxx` で書く。
- 大きいテーブルはincremental化・パーティション/シャード絞り込みでスキャン量を意識する。
- 命名規約（prefix等）はプロジェクトの既存実装に合わせる。

## 音声入力の読み替え
- ユーザーは音声入力を使うことがある。カタカナ化した技術用語や誤変換（例:「プレオペレーションズ」= pre_operations、「ディフィニションズ」= definitions/）は、文脈とリポジトリ内の実際の名前から正しい用語に読み替えて解釈する。読み替えに確信が持てない場合だけ、解釈を一言添えて確認する。
