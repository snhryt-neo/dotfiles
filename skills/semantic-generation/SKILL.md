# Semantic Generation

語を先に作って説明を後付けせず、指示対象を確定してから名前を与える。

## 1. 適用範囲を決める

次の成果物では対応表を作る。

- 設計文書、調査報告、原因分析、原因切り分け案、対策案
- 新しい状態名、条件名、事象名、値名、記録名、型名、メソッド名、boolean 名
- 複数段階の推論や処理を短い見出し、概念、作業ラベルへ要約したもの

軽微なコード編集や短い回答では、曖昧な新語を導入しそうな場合だけ適用する。

## 2. 本文より先に対応表を作る

次の列順を変えず、まず `候補語` と `初出定義` を空欄にして行を作る。

| 出典 | 目的 | 具体対象 | 役割 | 前後関係 | 候補語 | 初出定義 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |

各列を次の基準で埋める。

- `出典`: ユーザーの要求、仕様、コード、調査結果など、対象を確認できる根拠を書く。
- `目的`: その対象を区別または記述する理由を書く。
- `具体対象`: 候補語を使わなくても指示対象が一意になる具体文を書く。
- `役割`: `開始条件`、`状態`、`事象`、`値`、`記録`、`主体`、`対象`、`目的`、`手段`、`制約`から選ぶ。該当しなければ役割を具体的に記述する。
- `前後関係`: 時系列、因果関係、依存関係を元の用語で書く。関係がなければ `なし` とする。
- `候補語`: 最初の五列を検証した後でのみ埋める。
- `初出定義`: 新語の場合だけ `X とは、...を指す` と書く。既存の確立した用語なら `既存用語` とする。

一つの行には一つの具体対象と一つの役割だけを書く。同じ候補語が複数の対象または役割を指す場合は行を分け、原則として別の語を割り当てる。

## 3. 成果物の規模に合わせて表を扱う

文書ファイルとして提出する設計、調査報告、原因分析、対策案では、本文を書く前に環境が提供する一時ディレクトリへ `referent-table-<slug>.md` を保存する。本文だけを求められている場合は、この表を成果物のディレクトリへ残したり、勝手に本文へ埋め込んだりしない。

ユーザーが対応表を独立成果物として求めた場合、または監査要件で保存が必要な場合だけ、指定された保存先へ対応表を残す。

短い回答やコード内の局所的な命名では、対応表を作業中の判断材料として使い、ユーザーが求めていないファイルをリポジトリへ残さない。

SHA-256 は表の内容を識別できるが、表を本文より先に作ったこと自体は証明しない。監査要件またはユーザーの明示的な要求がある場合だけ記録する。

## 4. 候補語を決める

候補語は次の優先順位で選ぶ。

1. ユーザーが使っている語
2. コードベース、仕様、対象分野で確立している語
3. 具体対象と役割を過不足なく表す新語

新語の初出定義を書けない場合は命名せず、具体対象をそのまま記述する。目的・対象・判断を省いた抽象的な作業ラベルを、見出しや設計要素として導入しない。

## 5. 本文と実装へ写す

本文、設計要素、コード識別子の対応を保つ。対応表にない意味を候補語へ追加しない。

例えば、次の四つは別の役割なので一語にまとめない。

| 具体対象 | 役割 | 候補語の例 |
| :--- | :--- | :--- |
| 自動要約を始める境界として設定された会話履歴量 | 値 | `summaryStartThreshold` |
| 会話履歴量が自動要約の開始境界に達したこと | 開始条件 | `isSummaryStartThresholdReached` |
| 自動要約が始まること | 事象 | `automaticSummaryStarted` |
| 自動要約を実行していること | 状態 | `isAutomaticSummaryRunning` |

## 6. 矛盾を見つけたら表から直す

本文を書いている途中で指示対象や役割の混同に気付いたら、先に対応表の該当行を修正し、その行に依存する本文または識別子を更新する。説明の追記だけで矛盾を覆わない。
